    • Dispatch Table in One Line — Fixed Map with C++26 Reflection

    After years of design, elaboration, and experimentation, from Reflection TS to p2996, compile-time reflection has finally been adopted as one of the most significant C++26 features. Using the opaque, consteval-only type std::meta::info, developers can obtain, explore, or even define the meta-information of C++ entities — variables, functions, classes, class members, enum entries, type aliases, templates, and more — all during compile-time evaluation. This means that accessing the type, identifier, member list, or advanced properties like visibility and linkage of C++ entities no longer requires compiler magic or external code generators. Even more powerful is the ability to promote constexpr-evaluated data — such as a std::vector — to static storage via std::define_static_array(), std::define_static_string(), etc., offering a friendlier way to generate sequences of constants at compile time.

    In this post we will demonstrate the power of C++26 reflection with a common scenario: creating a fixed map — a series of key-value pairs whose data and layout are fixed at compile time — with the most efficient data structure and as little boilerplate as possible. Such fixed maps are frequently used in enum-to-string, string-to-enum, and function dispatch table scenarios. We will show how data is collected and promoted to static storage, and how C++26’s reflection enables automatic selection of the optimal data structure.

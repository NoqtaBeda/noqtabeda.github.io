    • Dispatch Table in One Line — Fixed Map with C++26 Reflection

    After years of design, elaboration, and experimentation, from Reflection TS to p2996, compile-time reflection has finally been adopted as one of the most significant C++26 features. Developers can obtain, explore, or even define the meta-information of C++ entities (variables, functions, classes, class members, enum entries, type aliases, templates, etc.) using the opaque, consteval-only type std::meta::info during compile-time evaluation. This means that accessing the type, identifier, member list, or advanced properties like visibility and linkage of C++ entities no longer requires compiler magic or external code generators. Even more powerful is the ability to promote constexpr-evaluated data — such as a std::vector — to static storage via std::define_static_array(), std::define_static_string(), etc., offering a friendlier way to generate sequences of constants at compile time.

    In this post we will demonstrate the power of C++26 reflection with a common scenario: creating a fixed map — a series of key-value pairs whose data and layout are fixed at compile time — with the most efficient data structure and as little boilerplate as possible. Such fixed maps are frequently used in enum-to-string, string-to-enum, and function dispatch table scenarios. We will show how data is collected and promoted to static storage, and how C++26’s reflection enables automatic selection of the optimal data structure.

    Background: Fixed Map Creation in Pre-C++26 Era

    Before C++26 reflection, building a fixed map — like dispatch_table in the example above — was a painful experience for C++ coders, requiring significant manual effort. A long if-else chain is straightforward, but it’s lengthy and bug-prone in practice, and becomes especially disastrous when the number of candidates is huge. Code generation macros may be helpful to some extent, yet blocks of boilerplate code are still required and readability may be harmed by macro syntax.

    • Extreme boilerplate: The code is extremely verbose.
    • Poor performance: Each lookup is O(n) — it compares against every entry sequentially until a match is found.
    • Maintenance nightmare: When the list grows to dozens or hundreds of entries, the function becomes unreadable and error-prone. A missing entry or a typo (have you noticed the typo above?) in a string literal may only surface at runtime.
    • Even AI is not 100% reliable: LLMs may also introduce random typos that are hard to spot in a long list — I tried generating the if-else chain and “shr” was mysteriously misspelled as “scr”.

    • Readability concern: Boilerplate code like #define OP_CASE and #undef OP_CASE is still required, and macros are harder to read than plain C++ code. If one operation naturally belongs to multiple different dispatch tables, the X macro must be either duplicated or decomposed (like OP_BIT_FUNCS as subset of OP_FUNCS in the example above), which involves more boilerplate code harming readability, and becomes even more troublesome when the dispatching policy is diverse.
    • Cryptic error messages: Any typo in the X macro or a forgotten #undef may produce deeply cryptic error messages from the preprocessor and compiler, often pointing to the wrong line.
    • Still poor performance: The generated code is still an O(n) linear chain.

    Approach 3: Runtime Map Creation

    To solve the time complexity issue, a common practice is to generate the fixed map during runtime, storing the generated data structure as a static variable:

    Drawbacks:

    • Runtime overhead: The map is initialized at runtime (on the first function call), incurring dynamic allocations and hashing costs — paying a price at runtime for data that is known entirely at compile time. Additionally, C++11’s thread-safe static local initialization may introduce small but non-zero synchronization overhead, even though the map itself is read-only thereafter.
    • Not usable in constexpr context: The dispatch table is a runtime object. constexpr-ness has zero chance.
    • Static initialization order pitfalls: If build_dispatch_table() depends on another static object, the initialization order is fragile and may lead to hard-to-debug crashes.
    • Unsuitable for constrained environments: Dynamic allocation and exceptions (required by std::unordered_map) may be unavailable or undesirable in embedded or freestanding contexts.

    Summary

    We have seen three pre-C++26 approaches to building a fixed dispatch table, each with its own trade-offs:

    Approach Maintenance Performance constexpr Support Boilerplate
    if-else chain ❌ Fragile ❌ O(n) ✅ Naturally constexpr ❌ Massive
    X macro ⚠️ Moderate ❌ O(n) ✅ Naturally constexpr ⚠️ Reduced but still verbose
    Runtime map ⚠️ Moderate ✅ O(1) ❌ Runtime only ⚠️ Reduced but still verbose

    None of them satisfy all requirements. The if-else chain and X macro both suffer from O(n) lookup and require manual data enumeration. The runtime map achieves O(1) lookup but sacrifices constexpr-ness and introduces initialization overhead. All three share the same root problem: there is no way to obtain the member list of namespace ops in an automatic manner. Manually maintained list (regardless of its form) is always required.

    C++26 reflection is a game-changer fundamentally. As the Live Demo shows, the RBOX_FUNCTION_FIXED_MAP macro (backed by reflection APIs) introspects the namespace ops automatically, discovers all functions whose identifier match the pattern "do_*", strips the common prefix to derive keys, and selects the optimal data structure — all at compile time, with zero manual enumeration and zero boilerplate. The resulting dispatch table is a constexpr value that can be inspected, combined, and validated before the program ever runs.

    Prerequisites

    Please read p2996 for introduction of C++26 Reflection if you are a beginner to C++26 world. You need to know:

    • Reflection operator ^^ — produces a std::meta::info from any C++ entity (e.g., ^^int reflects the type int, ^^ops::do_add reflects the function). How a reflection is obtained and accessed in consteval evaluation, especially:
      • std::meta::template_of() and std::meta::template_arguments_of() to query a template or template instance;
      • std::meta::substitute() to create a template instance;
      • std::meta::reflect_constant() to wrap a value into std::meta::info;
      • std::meta::extract<T>() to extract the value whose type is known as T from std::meta::info
    • Splicing operator [: :] — the inverse of ^^: extracts a reflected entity back into C++ code (e.g., typename([: ^^int :]) x = 0;). Understanding how a reflection is spliced back is essential;
    • Requirements to non-type template parameters and what is a structural type.

    Step by Step

    In the following contents, we will explore the process to build a fixed map for the dispatch table in live demo step by step:

    1. Compile-time Data Promotion — How to transform constexpr-evaluated data (such as a std::vector of key-value pairs) into a constexpr static array using std::define_static_array.
    2. Building a Simple Dispatch Table — How to enumerate namespace members via std::meta::members_of(), extract pointers to the dispatched function via std::meta::extract<T>(), and assemble a sorted key-value array suitable for binary search.
    3. Dispatching Candidate Type — How to select the optimal underlying data structure (linear list, binary search, or hash table) based on the characteristics of the input data, using std::meta::reflect_constant() to unify disparate return types under std::meta::info.
    4. Refining API: Auto-deduced class T — How to deduce the target type T from the filtered members, and promote type information from std::meta::info to template arguments of helper functions through the substitute + extract idiom.

    Compile-time Data Promotion

    C++26 standard library adopts a series of powerful tools to promote a constexpr-evaluated range or string to static storage. These tools effectively simplify compile-time constant array or string initialization.

    • std::meta::reflect_constant_array(range) reflects the given range as an array and returns a std::meta::info;
    • std::meta::reflect_constant_string(range) reflects the given range as a null-terminated string literal and returns a std::meta::info;
    • std::define_static_array(range) promotes the given range to static storage and returns a std::span<const T> which refers to the promoted array;
    • std::define_static_string(range) promotes the given range to static storage and returns const CharT* which points to the promoted null-terminated string.

    Technically, std::define_static_array() is an encapsulation of std::meta::reflect_constant_array(): it extracts the reflected array data via std::meta::extract<const T*> where T is the range value type. std::define_static_string() and std::meta::reflect_constant_string() works in a similar manner, plus appending the trailing '\0' if needed.

    std::meta::reflect_constant_array() can be implemented without compiler magic. A variable template __fixed_array of array type is used internally. The array template can be instantiated via std::meta::substitute().

    ⚠️ Due to its nature of template instantiation, std::define_static_array() mandates the range value type to be structural. A common pitfall is attempting to promote an array of std::span, std::string_view, etc. which are not structural (due to private data members). rbox library enables promoting nested data structures with structural alternative types rbox::meta_span, rbox::meta_string_view, etc. rbox provides a comprehensive helper rbox::to_static_storage() which promotes nested data structures recursively.

    Building a Naive Dispatch Table

    Building a naive dispatch table is straightforward with the help of std::define_static_array(). Consider the following simplified API: Given the namespace ns whose members are to be traversed, the identifier prefix by which members are filtered, and target type T, make_dispatch_table() obtains the filtered member list, then builds the dispatch table whose underlying data structure is simply a linear list of key-value pairs. Note that Span, and StringView in the return type can not be std::span or std::string_view since they violate the structural requirements of std::define_static_array() (due to private data members). In the actual implementation of rbox library, their structural alternatives rbox::meta_span and rbox::meta_string_view are used instead.

    C++26 standard library provides std::meta::members_of() which returns all the direct members of given entity (class/union type, namespace, etc.) as a std::vector<std::meta::info>. Then we use std::meta::has_identifier() and std::meta::identifier_of() to test and obtain the identifier of each member. Don’t forget to test the existence of an entity’s identifier before invoking std::meta::identifier_of(), otherwise a compile-time exception may be thrown (by the way, C++26 supports constexpr throwing and catching exceptions).

    Then we need to solve a critical question: how do we extract the desired pointer (which is const T*) from each filtered member (which is std::meta::info)? C++26 provides two ways for reflection entity extraction. refl below represents any std::meta::info value:

    • Using splicing operator [: refl :]: Requires refl to be a constant expression, but in our case member is not.
    • Using std::meta::extract<T>(refl): Requires the type of reflected entity T to be known. Specifically, if refl is a reflection for a function of type F, std::meta::extract<F*>(refl) extracts the corresponding function pointer.

    Dispatching Candidate Type

    The dispatch table we’ve just built works but is not fine-grained. No underlying data structure remains suitable for 100% of use cases. Hash table performs the best in most cases, but it may be troublesome when input data has hash collision, and it has worse performance than a naive linear list when the number of entries is fairly small. Our goal is to implement an adaptive dispatch table which selects the best data structure automatically. The solution is to wrap each result in a uniform type via std::meta::reflect_constant(), which takes a value of any structural type and produces a std::meta::info reflection. Now the function’s return type becomes std::meta::info. At the call site, the splicing operator [: :] extracts the underlying value: [: make_dispatch_table(...) :]. The user does not need to care about which underlying data structure is selected — as long as all candidates share the same API, they are logically interchangeable.

    Refining API: Auto-deduced class T

    The dispatch table till now works fine: given class T which is the target type, make_dispatch_table() obtains the filtered member list, builds the underlying data structure and then promotes it to static storage. But there’s one more thing we can improve. The explicitly specified target type T can be deduced from the filtered members with the help of std::meta::type_of(). Importantly, std::meta::extract<T*>() no longer works since T is now unknown.

    Now we face another critical problem. The deduced target type rT is not a constant expression, so how do we create the dispatch table object when T is unknown? The key is still std::meta::extract(), but this time we extract a pointer to a helper function instead. In general, the following pattern works as an idiom of C++26 reflection. The helper function make_something_impl<T>() is instantiated via std::meta::substitute(^^make_something_impl, {rT}); then std::meta::extract<ImplFunc*>(impl_instance) retrieves a function pointer to that instantiation, which we invoke indirectly. This idiom “promotes” the type information of T from std::meta::info to template arguments, enabling further constant-expression operations that depend on T. Note that in this pattern std::meta::reflect_constant() is needed to unify the return type, since std::meta::extract<ImplFunc*>() requires a consistent call signature ImplFunc.

    Now we get a much cleaner API for our demo: The target type int(int, int) can be eliminated.

    Conclusion

    C++26 reflection is not merely a language feature — it is a paradigm shift. C++26 reflection fundamentally transforms the way we build compile-time data structures. In this post we walked through the entire pipeline — from collecting entity information via std::meta::members_of, to promoting constexpr-evaluated data into static storage with std::define_static_array, to unifying disparate return types under std::meta::reflect_constant — culminating in a fixed map that requires zero manual enumeration and zero boilerplate.
